Introduction to Requirements Analysis

üìã Why Requirements Matter

Requirements analysis is an essential process in software development. It helps determine whether a system or project will meet its objectives and achieve success. Without proper requirements analysis, projects risk scope creep, missed deadlines, and ultimately, failure to deliver what stakeholders need.

Key Concept

To make requirements analysis effective, requirements are generally divided into two categories: Functional Requirements (what the system does) and Non-Functional Requirements (how the system performs).

02

Functional Requirements

Definition

Functional requirements define the specific features and operations a system must perform to meet business and user needs. They describe what the system should do and how it should interact with users or other systems.

üéØ Key Characteristics
  • Focus on System Behavior
    Functional requirements concentrate on what actions the system performs and how it behaves in response to inputs.
  • Observable and Testable
    These features can be directly observed and tested in the final product, making verification straightforward.
  • Common Examples
    User authentication, data processing, search functionality, payment handling, and report generation.
üí° Sample Questions to Identify Functional Requirements
What features should the system include?
What edge cases must be considered in the design?
How should the system respond to specific user inputs?
What data should the system store and retrieve?

Non-Functional Requirements

Definition

Non-functional requirements (NFRs) define how a system should operate, focusing on performance, reliability, and user experience rather than specific features. They ensure the system is efficient, secure, and maintainable over time.

üöÄ
Performance
Speed and responsiveness of the system under various conditions and load levels.
üîí
Security
Protection against unauthorized access, data breaches, and cyber threats.

Ease of use and intuitiveness of the interface for end users.
üõ°Ô∏è

üìà

Ability to handle growth in users, data, or transactions.
üîß
Maintainability

üåê
Portability
Ability to run in different environments and platforms.
üí° Sample Questions to Identify Non-Functional Requirements

How secure should it be against unauthorized access?
How available and reliable should the system be?
How many concurrent users must it support?

Real-World Examples

üè¶ Online Banking System
Functional Requirements
  • Users should be able to log in with their username and password
  • Users should be able to check their account balance
  • Users should receive notifications after making a transaction
Non-Functional Requirements
  • The system should respond to user actions in less than 2 seconds
  • All transactions must be encrypted and comply with industry security standards
  • The system should handle 100 million users with minimal downtime
üçî Food Delivery App
Functional Requirements
  • Users can browse the menu and place an order
  • Users can make payments and track their orders in real time
Non-Functional Requirements
  • The app should load the restaurant menu in under 1 second
  • The system should support up to 50,000 concurrent orders during peak hours
  • The app should be easy to use for first-time users, with an intuitive interface

06

Importance of Balancing Both Requirements

Balancing functional and non-functional requirements ensures a system is both useful and reliable. Focusing only on features without considering performance, security, or usability leads to systems that work technically but fail in practice.

01

Improves User Experience

Functional features alone may work but feel slow or hard to use. Non-functional aspects like performance, usability, and availability shape how users interact with the system.

02

Enhances System Performance

Scalability, reliability, and security ensure the system performs effectively under real-world conditions and varying loads.

03

Prevents Bottlenecks & Failures

Proper attention to non-functional needs reduces outages, security breaches, and downtime that can cripple a business.

04

Reduces Long-Term Costs

Addressing non-functional requirements early avoids expensive redesign, rework, or performance fixes later in the project lifecycle.

05

Supports System Evolution

Maintainability and extensibility allow the system to scale and adapt smoothly as business and user needs grow over time.

Common Challenges in Defining Requirements

Defining functional and non-functional requirements can be complex due to several challenges that teams must navigate carefully.

‚ö†Ô∏è Ambiguity in Requirements
Vague or incomplete requirements make it difficult to define what the system must do (functional) and how it should perform (non-functional). Clear, specific language is essential.
üîÑ Changing Requirements
Evolving business goals, market trends, or user expectations can shift requirements, making it harder to maintain stable design and development timelines.
üìä Difficulty in Prioritization
Functional requirements often take precedence, while critical non-functional aspects like security or scalability may be overlooked until it's too late.
üìè Measuring Non-Functional Requirements
Functional needs are easier to test; non-functional attributes like usability, scalability, or reliability are harder to quantify and validate objectively.
‚öîÔ∏è Overlapping or Conflicting Requirements
Some requirements may conflict‚Äîfor example, stronger security potentially impacting system performance‚Äîrequiring careful trade-offs and stakeholder discussions.

Extended Learning

As business students learning web technologies, understanding requirements is crucial because:

  • Bridge Between Business and Technology
    You'll often serve as the translator between what business stakeholders want and what developers build.
  • Project Success Depends on It
    Poor requirements are the #1 cause of project failure. Getting this right saves time and money.
  • Vendor and Contractor Management
    When outsourcing development, clear requirements protect your organization from scope creep and disputes.
Important Distinction

A requirement states WHAT is needed. An implementation describes HOW to achieve it. Keep these separate in your documentation!

‚úì REQUIREMENT
"Users must be able to search for products by name"
‚úó IMPLEMENTATION
"Use Elasticsearch with a React autocomplete component"

The IEEE 830 standard provides guidelines for writing Software Requirements Specifications (SRS). Key qualities of good requirements include:

  • Correct
    Every requirement accurately reflects a real need
  • Unambiguous
    Each requirement has only one interpretation
  • Complete
    All necessary requirements are included
  • Consistent
    No requirement conflicts with another
  • Verifiable
    Each requirement can be tested or checked
  • Modifiable
    Structure allows for easy changes
  • Traceable
    Origin of each requirement is clear
