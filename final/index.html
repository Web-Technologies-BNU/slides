‚öôÔ∏è Software Development Life Cycle (SDLC)

‚ñº
DEFINITION: SDLC is a structured framework/process for planning, creating, testing, and deploying software systems. It provides a systematic approach to software development that ensures quality and meets user requirements.

The Six Phases of SDLC

Requirements ‚Üí Design ‚Üí Implementation ‚Üí Testing ‚Üí Deployment ‚Üí Maintenance
01

Requirements Phase

Gather and document what the system should do. Interview stakeholders, analyze needs, define scope.

üìÑ SRS Document
02

Design Phase

Plan how the system will work. Create architecture, database schema, UI mockups, and wireframes.

üìê Wireframes, ERD
03

Implementation Phase

Write the actual code. Build frontend, backend, database, and integrate all components.

üíª Source Code
04

Testing Phase

Verify the system works correctly. Unit testing, integration testing, UAT, bug fixing.

üß™ Test Reports
05

Deployment Phase

Release the system to production. Server setup, data migration, user training, go-live.

üöÄ Live System
06

Maintenance Phase

Keep the system running. Bug fixes, updates, performance optimization, new features.

üîß Updates & Patches

üí° Exam Tip

The Requirements Phase is considered the most critical because errors here propagate throughout all subsequent phases. The cost of fixing requirements errors increases exponentially over time‚Äîfixing an issue in maintenance can cost 100x more than fixing it in requirements!

Why SDLC Matters

  • Reduces Risk: Systematic approach catches problems early
  • Improves Quality: Each phase has defined outputs and reviews
  • Controls Costs: Better planning prevents expensive rework
  • Ensures Documentation: Creates artifacts for future maintenance

üìã SRS & IEEE 830 Standard

‚ñº
DEFINITION: A Software Requirements Specification (SRS) is a formal document that describes what a software system should do. It serves as a contract between stakeholders and developers, following the IEEE 830 standard for requirements documentation.

Functional vs Non-Functional Requirements

Aspect Functional Requirements Non-Functional Requirements
Definition Define WHAT the system should do Define HOW the system should perform
Focus Specific behaviors and functions Quality attributes and constraints
Examples Login, search, checkout, send email Performance, security, usability, scalability
Testing Feature testing, user acceptance Load testing, security audits, benchmarks

Writing Good Requirements (IEEE 830)

Well-written requirements must have these characteristics:

  • Unambiguous: Only one possible interpretation
  • Complete: All requirements are documented
  • Verifiable: Can be tested/validated
  • Consistent: No contradictions between requirements
  • Traceable: Each requirement has a unique ID
  • Modifiable: Easy to update when needed

Common Mistake: Mixing Requirements with Implementation

‚ùå Wrong (Implementation Details)

"The system shall use React for the frontend and MySQL for the database."

"Use Bootstrap for responsive design."

"Implement REST API with Node.js."

‚úÖ Correct (Actual Requirements)

"The system shall allow users to search products by name, category, or price range."

"The system shall be accessible on devices with screen widths from 320px to 1920px."

"The system shall expose an API for third-party integration."

Converting Vague to Measurable Requirements

‚ùå Vague Requirement ‚úÖ Measurable Requirement
"The system should be fast" "Pages shall load within 3 seconds on 4G connections"
"It should be secure" "All data in transit shall be encrypted using TLS 1.2 or higher"
"Easy to use" "New users shall complete checkout within 5 clicks"
"Work on phones" "UI shall be fully functional on screens 320px and above"
"Handle many users" "System shall support 1000 concurrent users without degradation"

Standard Requirement Format

IEEE 830 Requirement Format 
// Functional Requirement Example
FR-01: The system shall allow users to reset their 
       password via email verification.

// Non-Functional Requirement Example
NFR-01: The system shall maintain 99.9% uptime 
        during business hours (9am-6pm).

// Key words:
"shall" = mandatory requirement
"should" = recommended but not mandatory
"may" = optional feature

üí° Exam Tip

When given a client statement, always separate technology choices (React, MySQL, Node.js) from actual requirements (what the user needs to do). Technology choices belong in the Design phase, not Requirements!

üìä Visitor Analytics & Heatmaps

‚ñº
DEFINITION: Visitor Analytics is the collection and analysis of data about website visitors‚Äîtheir behavior, demographics, and interactions. Tools like Microsoft Clarity provide insights through heatmaps, session recordings, and metrics.

Types of Analytics Data

üñ±Ô∏è Click Heatmaps

Visual representation of where users click on a page. Hot colors (red/orange) = high activity. Cold colors (blue) = low activity.

üìú Scroll Heatmaps

Shows how far users scroll down a page. Identifies where users stop reading or lose interest.

üé• Session Recordings

Video playback of actual user sessions. See exactly how users navigate, where they struggle, and what they do.

üëÜ Move Heatmaps

Tracks cursor movement patterns. Shows where users hover and focus attention (even without clicking).

Microsoft Clarity Metrics

Metric What It Measures Why It Matters
Dead Clicks Clicks on non-interactive elements Users expect something to happen but nothing does
Rage Clicks Repeated rapid clicks in same area User frustration‚Äîsomething isn't working
Quick Backs User leaves page within seconds Page didn't match expectations
Scroll Depth How far users scroll on a page Is content engaging? Are CTAs visible?
Excessive Scrolling Rapid up/down scrolling User is confused or looking for something

Interpreting User Behavior

Observed Behavior Possible Issue Solution
Clicking non-clickable images Users expect interaction (zoom/gallery) Make images clickable, add visual affordance
Rapid scrolling up/down Confusion, looking for info Add navigation, better headers, sticky menu
Cart abandonment at payment Complex form, trust issues Simplify form, add trust badges, guest checkout
Low scroll depth Content above fold not engaging Improve headline, add visual hooks
Rage clicks on button Button not responding/slow Fix functionality, add loading indicator

Benefits of Analytics

  • Data-Driven Decisions: Make changes based on actual user behavior, not assumptions
  • UX Optimization: Find and fix usability issues before they impact conversions
  • A/B Testing Validation: Measure which design performs better
  • ROI Tracking: Connect design changes to business metrics

üí° Exam Tip

When analyzing session recordings, look for patterns across multiple users. A single user's behavior might be an outlier, but if 50% of users do the same thing (like clicking a non-clickable element), that's a design problem to fix!

üì± Responsive Web Design

‚ñº
DEFINITION: Responsive Web Design is an approach that makes web pages render well on all devices and screen sizes. It uses flexible layouts, flexible images, and CSS media queries to adapt the design.

The Viewport

The viewport is the visible area of a web page on a device screen‚Äîthe portion users can see without scrolling. Without proper viewport configuration, mobile browsers render pages at desktop width then scale down, making text tiny and unusable.

Essential Viewport Meta Tag 
<!-- ALWAYS include this in <head> -->
<meta name="viewport" 
      content="width=device-width, initial-scale=1.0">

/* What it does:
   - width=device-width: Sets viewport width to device width
   - initial-scale=1.0: Sets initial zoom level to 100%
   - Enables media queries to work correctly */

Mobile-First vs Desktop-First

üì± Mobile-First (Recommended)

Approach: Design for mobile screens first, then add features for larger screens.

Uses: min-width media queries

Philosophy: Progressive Enhancement‚Äîstart simple, add complexity

üñ•Ô∏è Desktop-First

Approach: Design for desktop first, then remove/adjust for smaller screens.

Uses: max-width media queries

Philosophy: Graceful Degradation‚Äîstart complex, simplify

Why Mobile-First is Recommended:

  • Most users browse on mobile devices
  • Forces focus on essential content first
  • Progressive enhancement is easier than removing features
  • Smaller CSS files (base styles are simpler)

CSS Media Queries

Mobile-First Media Queries 
/* Base styles (Mobile) - No media query needed */
.container {
    width: 100%;
    padding: 15px;
}

/* Tablet and up */
@media (min-width: 768px) {
    .container {
        width: 750px;
        margin: 0 auto;
    }
}

/* Desktop and up */
@media (min-width: 1024px) {
    .container {
        width: 960px;
    }
}
Desktop-First Media Queries 
/* Base styles (Desktop) */
.sidebar {
    width: 300px;
    display: block;
}

/* Tablet and below - hide sidebar */
@media (max-width: 767px) {
    .sidebar {
        display: none;
    }
}

/* Mobile - single column */
@media (max-width: 480px) {
    .grid {
        grid-template-columns: 1fr;
    }
}

üí° Exam Tip

Remember the syntax: @media (condition) { rules }. Use min-width for mobile-first (adding styles as screen grows) and max-width for desktop-first (removing styles as screen shrinks).
