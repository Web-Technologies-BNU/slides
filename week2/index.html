What is Semantic HTML?

Definition: Elements whose names describe their meaning/role in the document (not just how they look). Browsers, assistive tech, and search engines can infer structure and purpose.

Benefits: Better accessibility (screen readers announce roles), improved SEO (search engines parse sections), cleaner code and easier collaboration.

Core semantic elements (definitions + use)

  • <header> ‚Äî Introductory content for a page/section (logo, title, actions).
  • <nav> ‚Äî Major navigation blocks of links (primary/secondary menus).
  • <main> ‚Äî Unique main content of the document (one per page).
  • <section> ‚Äî Thematic grouping of content (with a heading).
  • <article> ‚Äî Self‚Äëcontained, reusable unit (post, card, press release).
  • <aside> ‚Äî Tangential/secondary content (sidebars, callouts).
  • <footer> ‚Äî Closing area for page/section (copyright, links).

Non‚Äësemantic elements (and when to use)

Definition: Elements whose names do not convey meaning (e.g., <div>, <span>). They are neutral containers used for grouping or styling when no semantic element fits.

  • <div> ‚Äî Block‚Äëlevel generic container (layout wrappers, utility groupings).
  • <span> ‚Äî Inline generic container (highlight text, apply a class).
Rule of thumb: Prefer a semantic element if one clearly describes the content. Use <div>/<span> to fill gaps or for hooks/classes.

Inline vs Block elements

Block examples

<h1>Heading</h1>
<p>Paragraph</p>
<ul><li>item</li></ul>

Start on a new line and take available width.

Inline examples

<em>emphasis</em> and <strong>importance</strong>
<a href="#">link</a> and <span class="tag">tag</span>

Flow within a line, no line break by default.
